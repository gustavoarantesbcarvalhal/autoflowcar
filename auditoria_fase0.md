## 3. Erros Encontrados

### 3.1 Erros de TypeScript

**[MÉDIO] `admin.tsx` — `PlanoBadge` declarada e nunca utilizada**
```
admin.tsx:47 — function PlanoBadge() { ... }
```
O componente `PlanoBadge` é definido mas `TenantRow` usa um `<select>` inline, não o badge. Vai gerar warning do ESLint `no-unused-vars`. O `PLANO_LABELS` também fica como dead code por consequência.

**[BAIXO] `AuthContext.tsx` — `loadPerfil` ausente nas dependências do `useEffect`**
```tsx
// linha 70 — loadPerfil não está no array de deps
useEffect(() => {
  supabase.auth.getSession().then(async ({ data: { session } }) => {
    if (session?.user) await loadPerfil(); // loadPerfil não está no dep array
    ...
  });
}, []); // ESLint react-hooks/exhaustive-deps vai reclamar
```
Não é um bug em runtime (porque o efeito só roda uma vez e `loadPerfil` fecha sobre `setPerfilData` que é estável), mas vai gerar warning de linting.

**[BAIXO] `admin.tsx` — Cast desnecessário**
```tsx
// linha 105
(tenant?.plano as "starter" | "pro" | "white_label") ?? "starter"
```
Como `tenant.plano` já é `Database["public"]["Enums"]["tenant_plano"]`, este cast é redundante mas não é erro.

### 3.2 Erros de Build / Importação

**[ALTO] `routeTree.gen.ts` foi editado manualmente**

O arquivo contém `// This file was automatically generated by TanStack Router. You should NOT make any changes.` A próxima execução de `bun dev` vai **sobrescrever** todas as edições manuais. Como o Vite plugin de TanStack Router detecta automaticamente os novos arquivos de rota, as rotas `/login` e `/admin` serão incluídas ao iniciar o dev server. O problema é que até o primeiro `bun dev`, o arquivo manual funciona — mas após o primeiro run, ele pode ser regravado. **As edições são descartáveis e funcionalmente corretas até a primeira execução do build.**

**[ALTO] Tipos TypeScript desatualizados para tabelas existentes**

A migração SQL adiciona `tenant_id UUID` às tabelas `customers`, `vehicles`, `appointments` e `interactions`, mas os tipos TypeScript dessas tabelas em `types.ts` não foram atualizados para incluir o campo. Isso significa:
- `supabase.from('customers').insert({ ..., tenant_id: '...' })` vai gerar **erro TypeScript** porque `tenant_id` não está no tipo `Insert` de `customers`.
- Qualquer código de fases futuras que tentar usar `tenant_id` nos tipos existentes falhará na checagem de tipos.

## 4. Riscos Encontrados

### 4.1 Críticos (bloqueantes para produção)

**[CRÍTICO] Server functions sem verificação de autenticação**

`criarTenantComAdmin` e `atualizarTenant` usam `supabaseAdmin` (service role, bypassa RLS) sem verificar se o chamador está autenticado ou é super_admin. Qualquer requisição HTTP POST ao endpoint das server functions pode criar ou modificar tenants sem credenciais.

```typescript
// admin.functions.ts — NÃO há middleware de autenticação
export const atualizarTenant = createServerFn({ method: "POST" })
  .inputValidator(...)
  .handler(async ({ data }) => {
    const { supabaseAdmin } = await import("@/integrations/supabase/client.server");
    // service role — bypassa RLS — SEM checagem de quem está chamando
    await supabaseAdmin.from("tenants").update(updates).eq("id", id);
  });
```

O middleware `requireSupabaseAuth` existe em `src/integrations/supabase/auth-middleware.ts` mas não foi utilizado nas server functions.

**[CRÍTICO] `SUPABASE_SERVICE_ROLE_KEY` ausente no `.env`**

O arquivo `.env` atual tem apenas `SUPABASE_PUBLISHABLE_KEY`. Sem a `SUPABASE_SERVICE_ROLE_KEY`, o `supabaseAdmin` lança erro em runtime e ambas as server functions falham completamente — impossível criar ou editar lojas pelo painel.

**[CRÍTICO] Tipos TypeScript desatualizados para tabelas existentes**

Conforme descrito em 3.2: `customers`, `vehicles`, `appointments` e `interactions` têm `tenant_id` no banco (após a migração) mas não nos tipos TypeScript. Todo o código de fases futuras que tentar inserir dados com `tenant_id` vai falhar em tempo de compilação.

### 4.2 Altos (impacto direto no usuário)

**[ALTO] Flash de `BlockedScreen` para usuários com perfil carregando**

Em `__root.tsx` `AppShell`:
```tsx
if (user && !temAcesso) {
  return <BlockedScreen />;
}
```
`temAcesso` é `false` quando `perfilData` é `null`. Durante o intervalo entre o `signIn` ser concluído (`user` fica definido) e o `loadPerfil()` terminar (`perfilData` fica populado), há uma janela onde `user !== null && perfilData === null`, portanto `temAcesso === false`. Isso pode fazer a tela de "Acesso bloqueado" piscar brevemente para **todos** os usuários legítimos ao fazer login.

**[ALTO] Usuário bloqueado sem forma de fazer logout na tela de login**

Se um usuário bloqueado faz login:
1. `signIn()` roda com sucesso (Supabase não bloqueia no auth layer)
2. `loadPerfil()` retorna `tenant_status: "bloqueado"`
3. `temAcesso = false` → usuário não é redirecionado para `/`
4. `isBlocked = true` → formulário de login some, aparece banner de bloqueio
5. **Não há botão de logout na tela de login para usuários bloqueados**

O usuário fica preso na tela de login sem conseguir sair da conta.

**[ALTO] Race condition entre `getSession` e `onAuthStateChange`**

```tsx
// AuthContext.tsx:70
supabase.auth.getSession().then(async ({ data: { session } }) => {
  setUser(session?.user ?? null);
  if (session?.user) await loadPerfil();
  setLoading(false); // ← pode ser chamado DEPOIS do onAuthStateChange abaixo
});

supabase.auth.onAuthStateChange(async (_event, session) => {
  setUser(session?.user ?? null);
  // ...
  setLoading(false); // ← pode ser chamado ANTES do getSession acima
});
```

Se `onAuthStateChange` disparar (ex: `INITIAL_SESSION`) antes do `getSession().then()` resolver, `setLoading(false)` é chamado duas vezes com estados potencialmente inconsistentes entre as duas calls.

### 4.3 Médios

**[MÉDIO] Guard de auth duplicado e potencialmente conflitante**

Há guards de auth em três lugares:
- `AppShell` em `__root.tsx` (useEffect)
- `LoginPage` em `login.tsx` (useEffect)
- `AdminPage` em `admin.tsx` (useEffect)

Quando o usuário faz login, todos os três useEffects podem disparar ao mesmo tempo com múltiplas chamadas a `navigate()` simultâneas. TanStack Router geralmente é tolerante a isso, mas pode gerar navegações redundantes.

**[MÉDIO] `criar_tenant()` SQL existente mas nunca chamado**

A função SQL `criar_tenant()` foi criada na migração mas a aplicação usa exclusivamente a server function `criarTenantComAdmin`. A função SQL é inutilizada — o GRANT de `EXECUTE` foi concedido desnecessariamente ao role `authenticated`, o que é uma surface de ataque sem utilidade.

**[MÉDIO] Nenhuma transação atômica na criação de tenant**

`criarTenantComAdmin` faz três operações sequenciais: criar auth user → criar tenant → criar profile. O rollback manual em caso de falha parcial (passos 2 ou 3) pode falhar silenciosamente:
```typescript
// Se deleteUser falhar, o usuário fica órfão no Supabase Auth
await supabaseAdmin.auth.admin.deleteUser(authUser.user.id);
await supabaseAdmin.from("tenants").delete().eq("id", tenant.id);
```
Se o cleanup falhar, o banco fica em estado inconsistente (auth user sem tenant, ou tenant sem profile).

**[MÉDIO] Dados existentes órfãos**

Toda a data existente nas tabelas `customers`, `vehicles`, `appointments`, `interactions` tem `tenant_id = NULL` após a migração. Com a nova RLS, esses dados são invisíveis para qualquer usuário autenticado (exceto super_admin). A demo data existente em produção fica efetivamente perdida sem uma migração de atribuição a um tenant.

**[MÉDIO] Modal de edição de loja incompleto**

O spec menciona "Modal para editar loja existente". A implementação inclui apenas `nome`, `email_admin` e `plano`. Os campos `logo_url` e `cor_primaria` (definidos no schema) não aparecem no modal de edição.

### 4.4 Baixos

**[BAIXO] `PlanoBadge` é código morto em `admin.tsx`**

O componente é definido mas nunca renderizado. O `PLANO_LABELS` também fica sem uso real.

**[BAIXO] Sem isolamento por usuário dentro do tenant**

O spec define que `vendedor` vê apenas seus próprios leads. A RLS atual implementa isolamento apenas por tenant, não por usuário. Todos os perfis de uma loja (admin, gerente, vendedor) enxergam todos os customers do tenant.

**[BAIXO] Sem `UNIQUE` constraint em `tenants.email_admin`**

Possível criar dois tenants com o mesmo email de admin. Não causa erro de runtime mas dificulta lookup.

**[BAIXO] `isPublicPage` é alias redundante de `isLoginPage`**

```tsx
const isLoginPage = pathname === "/login";
const isPublicPage = isLoginPage; // sempre idêntico
```

`isPublicPage` foi definido provavelmente para acomodar rotas públicas futuras, mas atualmente é um alias desnecessário.

**[BAIXO] Métricas calculadas no cliente**

Total de lojas, ativas, pro e white-label são calculados filtrando o array in-memory. Para grandes volumes (1000+ tenants), seria mais eficiente uma query agregada no banco.

O maior problema objetivo é a **ausência de autenticação nas server functions** (crítico para produção) e a falta da **service role key** no ambiente. Os outros itens são correções pontuais de baixo esforço. Após resolver os 5 pontos obrigatórios da seção anterior, o sistema estará pronto para iniciar a Fase 1.
